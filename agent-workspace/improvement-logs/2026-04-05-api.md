# API Agent Improvements — 2026-04-05

## Changes Made (8 files)

### 1. Added `withRateLimit("read")` to unprotected public GET routes
**Files:** `traffic/route.ts`, `photos/route.ts`, `similar/route.ts`, `favorites/ids/route.ts`, `admin/image-search/route.ts`
**Why:** These public GET endpoints had no rate limiting, allowing unlimited requests per IP. All other public GETs already used `withRateLimit`.

### 2. Added `withRateLimit` to notifications route
**File:** `src/app/api/notifications/route.ts`
**Why:** GET (read) and POST (write) were unprotected. Also added try/catch error handling that was missing.

### 3. Added UUID validation to menu route
**File:** `src/app/api/restaurants/[id]/menu/route.ts`
**Why:** Restaurant ID param was passed directly to Prisma without UUID format check. Other `[id]` routes already validated.

### 4. Migrated crawl/area route to consistent API envelope
**File:** `src/app/api/crawl/area/route.ts`
**Why:** Used raw `Response.json()` instead of `apiSuccess()`/`apiBadRequest()`/`apiError()`. Now consistent with all other routes.

### 5. Migrated admin/image-search to consistent API envelope
**File:** `src/app/api/admin/image-search/route.ts`
**Why:** Used raw `Response.json()` with no rate limiting. Now uses `withRateLimit("read")` + `apiSuccess()`/`apiBadRequest()`.

## Routes still using manual rate limiting (not `withRateLimit`)
These use `checkApiRateLimit()` directly because they need custom logic (auth cookie setting, etc.):
- `auth/login`, `auth/register` — auth tier
- `favorites/route.ts` POST — write tier  
- `users/profile` PATCH — write tier
- `crawl/restaurant` — crawl tier

These are acceptable since they still have rate limiting, just via the lower-level API.
