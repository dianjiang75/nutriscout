## Changes Made

### 1. Review Aggregator — Parallel dish summarization (review-aggregator/index.ts)
**Before**: Sequential `for...of dishes` loop — each `summarizeDishReviews()` LLM call waited for the previous. A 40-dish restaurant took ~40 serial Qwen API calls.
**After**: `Promise.allSettled` with concurrency=3 in batches. Speedup: ~3x.
**Why**: LLM summarization calls are I/O-bound. Parallelism up to rate limit is pure win. Failed dishes log a warning and are skipped, not crash the batch.

### 2. Apollo Evaluator — EU 14 allergen coverage (evaluator/index.ts)
**Before**: `ALLERGEN_KEYWORDS` covered FDA Big 9 only.
**After**: Added EU 14 allergens not in FDA set:
- `celery`: celery, celeriac, celery root, celery salt, celery seed, lovage
- `mustard`: mustard, mustard seed, mustard oil, dijon, whole-grain mustard
- `lupin`: lupin, lupine, lupin flour, lupin seed, lupin bean
- `sulphites`: sulphite, sulfite, sulphur dioxide, dried fruit, wine sauce, pickled
- `molluscs`: snail, escargot, abalone, squid ink, cuttlefish ink
Also expanded `fish` keywords to include worcestershire (contains anchovies) and `tree_nuts` to include praline, nougat, marzipan, frangipane.
**Why**: EU visitors + restaurants increasingly need EU allergen compliance. Sulphite allergy is common and commonly missed.

### 3. Apollo Evaluator — Expanded KNOWN_ALLERGEN_DISHES (evaluator/index.ts)
**Before**: ~80 entries covering obvious hidden allergen dishes.
**After**: +40 entries:
- `nut_free`: Thai curry variants, massaman, panang, praline, romesco, muhammara, pecan pie
- `gluten_free`: couscous, bulgur, teff, orzo, beer-battered, tempura, katsu, teriyaki/hoisin/oyster sauce (wheat-based soy), miso soup, ramen broth
- `dairy_free`: white sauce, béchamel, hollandaise, béarnaise, risotto, mashed potato, chowder, lobster bisque, creamy soups
- `vegan`: caesar dressing, worcestershire, dashi, bonito flakes, katsuobushi
- `vegetarian`: caesar salad, ramen/pho (meat broth), french onion soup (beef broth), paella
**Why**: Hidden allergens in these dishes are the primary cause of false safety flags. More entries = stricter evaluation = safer users.

## Regressions Found & Fixed
- Vision `responseSchema` as `const X = {...} as const` caused TypeScript error because `required: string[]` became `readonly tuple` which was incompatible with Gemini SDK's mutable `string[]` type. Fixed by using explicit `Schema` type annotation instead of `as const`.

### 7. Search Orchestrator — Cuisine diversity cap (orchestrator/index.ts)
**Before**: `applyRestaurantDiversityCap()` capped at max 3 dishes per restaurant only.
**After**: Added per-cuisine cap (default max 6 per cuisine). When searching "pasta", results won't be exclusively Italian — Mediterranean, fusion, and other cuisines that serve pasta variants are included. The cap is higher than the restaurant cap (6 vs 3) because cuisine is a broader grouping.
**Why**: Narrow search results from one cuisine provide poor UX for discovery. Users want to find the best pasta anywhere, not just from top-ranked Italian restaurants.

### 8. USDA Client — includeBranded parameter (usda/client.ts)
**Before**: `searchFood()` only queried `Foundation,SR Legacy` data types.
**After**: Added optional `includeBranded = false` parameter. When `true`, includes USDA's Branded data type (1M+ products including McDonald's, Chipotle, Sweetgreen, etc.). Default stays false to preserve data quality for generic ingredients.
**Why**: For chain restaurants, USDA Branded data is more accurate than generic entries (e.g., "McDonald's Filet-O-Fish" has known exact macros). Caller can opt in when restaurant context is known.

### 9. page.tsx — startTransition for sort/category updates (INP fix)
**Before**: Sort button clicks and category pill selections synchronously updated React state, blocking the main thread before the skeleton could appear. This caused the interaction to feel sluggish (INP >200ms on mid-range devices).
**After**: Wrapped `setSearch` calls in `startTransition()` for category and sort state updates. React treats these as non-urgent — the button press registers instantly, state update is deferred.
**Why**: React 18+ `startTransition` is the canonical fix for INP (Interaction to Next Paint). The threshold is 200ms; filter/sort changes triggered full re-renders that could block for 100-300ms.

### 10. globals.css — scroll-padding-top for sticky header (WCAG 2.2 criterion 2.4.11)
**Before**: Keyboard navigation to off-screen elements would scroll the element to the top of the viewport, where the sticky header covered it.
**After**: `scroll-padding-top: 4rem` added to `html` block — browser adds this offset when scrolling focused elements into view, ensuring they appear below the sticky header.
**Why**: WCAG 2.2 criterion 2.4.11 (Scrolling Target in Viewport): keyboard users must be able to see focused elements without them being obscured by sticky UI. Zero visual change.
