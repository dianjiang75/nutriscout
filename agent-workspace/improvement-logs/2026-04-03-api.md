# API Agent Report — 2026-04-03

## Audit Results

Audited all 28 API routes. Most routes already have good protection:
- **18 routes** use apiSuccess/apiError helpers
- **16 routes** have Zod validation
- **13 routes** have rate limiting
- Auth routes (login, register) have dedicated "auth" rate limits (5/min)
- Search has "search" rate limits (60/min)
- Crawl endpoints have "crawl" rate limits (5/min)

## Changes Applied

### 1. Favorites Routes — Standardized to apiSuccess/apiError (MEDIUM)
**Files**: `src/app/api/favorites/route.ts`, `src/app/api/favorites/ids/route.ts`
**Problem**: Both routes used raw `Response.json()` with inconsistent error shapes. The GET, POST, and toggle endpoints all returned different formats.
**Fix**: Migrated to `apiSuccess()`, `apiUnauthorized()`, `apiBadRequest()`, `apiRateLimited()`, `apiError()` helpers. Now all favorites responses follow the `{ success: true, data: ... }` / `{ success: false, error: ... }` envelope.

### 2. Notification Preferences — Added Rate Limiting (MEDIUM)
**File**: `src/app/api/notifications/preferences/route.ts`
**Problem**: GET and PUT endpoints had no rate limiting despite being authenticated endpoints. An attacker with a stolen token could spam preference changes.
**Fix**: Wrapped both handlers with `withRateLimit`: GET uses "read" (100/min), PUT uses "write" (20/min).

### 3. Restaurant Traffic — Added UUID Validation (LOW)
**File**: `src/app/api/restaurants/[id]/traffic/route.ts`
**Problem**: No validation on the `id` path parameter. Malformed IDs would hit Prisma and throw cryptic database errors.
**Fix**: Added UUID regex validation before any DB query. Returns `apiBadRequest("Invalid restaurant ID")` for non-UUID paths.
