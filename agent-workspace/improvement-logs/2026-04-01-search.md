# Search Agent Report — 2026-04-01

## Changes Made

### 1. Similarity engine geographic filtering (`src/lib/similarity/index.ts`)
- **Before**: `findSimilarDishes()` returned dishes from any restaurant globally
- **After**: Filters candidates by `options.radius_miles` using haversine distance
- **Impact**: Similar dish suggestions are now geographically relevant

### 2. `macro_match` sort implemented (`src/lib/orchestrator/index.ts`)
- **Before**: `macro_match` sort fell through to nutritional goal ordering via DB ORDER BY
- **After**: New `macroMatchScore()` function does in-memory scoring based on the user's nutritional goal (max_protein, min_calories, min_fat, min_carbs, balanced)
- **Impact**: Users can sort by how well dishes match their macro targets

### 3. `wait_time` sort fixed (`src/lib/orchestrator/index.ts`)
- **Before**: `wait_time` sort used `createdAt: "asc"` as DB ORDER BY (meaningless)
- **After**: DB fetches with `createdAt: desc`, then in-memory sort by actual `estimated_wait_minutes`
- **Impact**: Wait time sorting now works correctly

### 4. Cache key includes search text and categories (`src/lib/cache/index.ts`)
- **Before**: Cache key was `query:<filters>:<goal>:<lat>,<lng>:r<radius>` — different search terms at same location returned same cached result
- **After**: Key includes `searchText`, `categories`, `sortBy` 
- **Impact**: Cache actually differentiates between "pad thai" and "pizza" searches
- Tests updated to match new cache key format

### 5. Max wait time filter (`src/lib/orchestrator/index.ts`)
- **Before**: `max_wait_minutes` field existed on UserSearchQuery but was never used
- **After**: Filters out dishes at restaurants exceeding the wait threshold
- **Impact**: Users can exclude restaurants with long waits

## Validation
- `npx tsc --noEmit`: PASS (0 errors)

## Still Needs Attention
- Full-text search via tsvector requires the `search_vector` generated column + GIN index in DB (raw SQL migration needed by backend agent)
- `distance` sort works via in-memory haversine — for large datasets, should use PostGIS `earthdistance` at DB level
- Dietary filter still uses hard AND logic (user wanting vegan OR gluten_free can't do that)
- `category` filter uses `mode: "insensitive"` — Prisma may not support this on string arrays
