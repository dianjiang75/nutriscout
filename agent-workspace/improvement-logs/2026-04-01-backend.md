# Backend Agent Report — 2026-04-01

## Research Summary
- Prisma connection pooling: singleton pattern + `PrismaPg({ max: N })` for pool size
- PostgreSQL indexes: GIN for JSONB/tsvector, B-tree for category/availability filters
- BullMQ deduplication: custom `jobId` per place/dish prevents duplicate processing
- Redis health: `PING` check at startup; `lazyConnect` already configured

## Changes Made

### 1. Database indexes (`prisma/schema.prisma`)
- Added `idx_dishes_category` on `category` for category-based filtering
- Added `idx_dishes_available` on `isAvailable` for availability queries
- Added `idx_dishes_macro_source` on `macroSource` for pipeline status queries
- **Impact**: Faster queries for search, category filtering, pipeline monitoring

### 2. Redis health check (`src/lib/cache/redis.ts`)
- Added `checkRedisHealth()` function returning `{ ok, latencyMs }`
- Can be used in health check endpoints and startup validation
- **Impact**: Detects Redis failures at startup instead of first user request

### 3. Cache invalidation optimization (`src/lib/cache/index.ts`)
- **Before**: `invalidateRestaurant()` scanned ALL keys with `*${id}*` pattern
- **After**: Scans only `restaurant:${id}:*` prefix, much narrower keyspace
- Also invalidates query cache since search results may reference the restaurant
- **Impact**: O(n) on restaurant keys vs O(N) on all keys

### 4. Worker singleton Prisma clients (`workers/crawl-worker.ts`, `workers/photo-worker.ts`)
- **Before**: Crawl worker's completed handler created a new PrismaClient per job
- **After**: Both workers use module-level singleton PrismaClient with pool config
- **Impact**: Prevents connection exhaustion under load

### 5. Job deduplication (`workers/crawl-worker.ts`)
- Added `jobId: photo-${dish.id}` to photo analysis jobs
- Documented deduplication pattern: `jobId: crawl-${googlePlaceId}` for crawl jobs
- **Impact**: Same restaurant/dish can't be processed simultaneously

### 6. Photo worker started in start-all.ts (`workers/start-all.ts`)
- Photo worker was defined but never started — now imported in start-all.ts
- **Impact**: Photo analysis actually runs when workers are launched
