# Pipeline Agent Improvement Log — 2026-04-05

## Session: Pipeline reliability fixes

### Fix 1: Photo analysis now queued via BullMQ (CRITICAL)
**File**: `src/lib/agents/menu-crawler/index.ts`
**Before**: `crawlRestaurant()` called `batchAnalyzePhotos()` inline (fire-and-forget). This bypassed the BullMQ queue entirely — photos were analyzed in-process without retries, rate limiting, or deduplication. The `photosQueued` counter in CrawlResult reported correct numbers but the jobs never went through the queue system.
**After**: Photos are now queued via `photoAnalysisQueue.addBulk()` with proper `jobId: photo-${dishId}` deduplication, priority 2, 2 retry attempts with exponential backoff. Uses `addBulk()` for a single Redis pipeline call instead of N round-trips. Queue instance created on-demand from `@/lib/cache/redis` and closed after use.
**Impact**: Photos will now be processed by `photo-worker.ts` with rate limiting (5/min), retries, dead letter queue on failure, and proper concurrency control.

### Fix 2: Failed ingredient batches no longer silently dropped
**File**: `src/lib/agents/menu-crawler/index.ts`
**Before**: When the outer try/catch in `analyzeIngredients()` caught an API error (network timeout, rate limit, etc.), the entire batch was silently dropped — no placeholder entries were created, so dishes from that batch would have no dietary analysis at all.
**After**: Failed batches now produce placeholder entries with `dietary_confidence: 0` and a warning message, matching the behavior of the inner JSON parse failure handler. This ensures all dishes are accounted for in the results array.

### Fix 3: Improved category detection in HTML menu parser
**File**: `src/lib/agents/menu-crawler/sources.ts`
**Before**: Category detection relied on `closestSection` and `prevAll("h2, h3")` which only searched immediate siblings. On flat menu HTML (common on restaurant sites), headings are often separated by multiple levels of nesting, so category was almost always null.
**After**: Added `data-category` attribute detection, deeper parent traversal (up to grandparent level for preceding headings), and a heading collection pass for structured menu sections. Category detection now covers more real-world HTML patterns.

### Not changed (already working)
- **Price parsing** (Issue #7): `parsePriceString()` already handles "Market Price", "$$$", ranges — no fix needed.
- **Review fuzzy matching** (Issue #4): `filterReviewsForDish()` already uses word boundaries (\b) and 85% threshold. Short words (<4 chars) like "Pad" are filtered from word matching and fall back to full-name exact match.
- **USDA matching** (Issue #2): `decomposeIngredientName()` + 418-entry `USDA_SYNONYMS` map + LLM fallback already handles compound names well. The pipeline is: synonym lookup -> cooking modifier stripping -> sub-phrase decomposition -> LLM-assisted USDA term suggestion.
